理解现有产品与用户处境

Understanding the Existing Product & User Context

AWS CID 基于 Amazon QuickSight 构建，包含 CUDOS、CID、KPI Tracker 三套仪表盘，覆盖从细粒度溯源到战略决策的完整成本运营场景。

人工界面审计 · 12 张截图标注
覆盖 Billing、KPI Tracker、Spot Savings、RI Summary 四个模块的逐屏标注审查
审计过程为逐屏人工标注，涵盖12张 CID 仪表盘截图。每张截图逐项检查：层级冲突、重复标签、导航断裂、颜色语义不一致——所有问题分类记录后，才进入重设计阶段。 Each screen was inspected for hierarchy conflicts, duplicated labels, navigation issues, and color semantic inconsistencies.
关于研究方法 — A Note on Method
审计过程中使用了 AI 工具辅助归纳——主要用于将多张截图中反复出现的界面问题（重复标签、层级冲突、颜色语义不一致）聚类整理。所有问题定义、设计决策与重设计方向，均在人工审查 CID 仪表盘与 AWS 文档后独立判断。 AI was used only as a synthesis aid; all design decisions were made manually. 交互原型从零独立搭建。
项目的真实起点

这个项目最初只是一次视觉审计练习。仪表盘内容密集、难以快速扫读，所以最初的判断是：主要问题出在视觉层级上——太多图表在同一屏内争夺注意力。

但反复审查多张 CID 截图后，我意识到更深的问题不是视觉密度，而是决策路径的缺失。用户能看到异常，但界面没有给出"下一步该做什么"的引导。从"发现问题"到"定位具体资源、确认成本影响、执行建议操作"，需要跨越多个断裂的视图手动操作。这个认知让设计重心从视觉整理转向了围绕决策流程重构信息架构。 The focus shifted from visual cleanup to restructuring the IA around a decision workflow.

👤

现有系统问题归纳
01
信息密度过载，无视线引导
QuickSight 等宽网格使所有数据权重相同。CUDOS 全局视图一屏堆叠 20+ 图表，用户无法判断从哪里开始看。
02
颜色语义不一致
荧光绿同时表示"Spot 节省"、"目标达成"、"值下降（省钱好）"三种含义，用户每次都需要重新解码颜色。
03
KPI 超标无行动路径
EC2 Prev Gen 达 43%（目标≤10%），与达标行视觉权重一致，用户看到问题却找不到下一步入口。
04
三套仪表盘割裂
CUDOS / CID / KPI 是独立 URL，无共享筛选器。发现异常后切换模块，账号过滤上下文完全丢失。
05
角色混用，无专属视图
CID 声称面向高管，但仍包含 Lambda 调用明细等技术数据，高管与工程师内容边界不清。
06
字体层级单薄
标题、数字、标签字重相近，深色背景下对比不足。RI 页标题用 32px 粗体，与内容卡片比例失调。
重设计 — 信息架构重构
03 · Information Architecture · 信息架构

重构信息架构

Restructuring the Information Architecture

从"按工具分三个独立页面"转向"按用户旅程分层"。三类角色共享统一入口与筛选器，按权限渐进展开内容，消除跨模块上下文丢失问题。

现有 IA vs 重设计 IA
🔴 现有系统 — 三套割裂的独立仪表盘
× CUDOS / CID / KPI 三个独立 URL，无统一导航
× 切换模块需重新设置筛选条件，上下文丢失
× 高管视图与工程师视图混用同一页面
× 没有引导"发现问题→定位根因→采取行动"的流程
🟢 重设计 — 统一平台 + 角色分层
✓ 统一左侧导航，全局筛选器跨页面共享
✓ Executive / FinOps / Engineering 三层视图
✓ 每个超标项旁边提供"下一步行动"入口
✓ 看见→理解→行动→追踪的完整决策旅程

设计系统与界面原型

Typography
Before: QuickSight default, no scale
After: Inter UI + JetBrains Mono for all numeric values, 5-level type scale
Color Semantics

重设计成果
05 · Outcome & Reflection · 结论

预期改善与设计反思

Projected Improvements & Design Reflection

基于界面审计建立的基准，重设计方案在四个核心体验维度预计产生显著改善。

预期改善方向 — Expected Impact
从异常发现到根因定位的操作步骤减少，通过内联展示推荐操作实现 — fewer steps from anomaly to action
统一颜色语义后的扫读效率提升——每种颜色只对应一个含义，无例外 — consistent color semantics
跨模块切换时筛选器状态全程保留，无需反复重设上下文 — persistent global filter context
高管总览与工程细节视图通过角色分层清晰分离 — role-based progressive disclosure
核心洞察
问题的根源不是视觉密度，而是界面对所有数据一视同仁——没有从"发现异常"到"下一步操作"的自然路径。重设计的本质是让这条决策路径变得显性可见。 Making the decision path explicit.
设计取舍
我没有尝试把所有 CID 分析视图合并到一个仪表盘——那样只会重新制造当前的信息过载问题。重设计优先保证跨视图的引导式导航，而非任意单屏的内容完整性。部分细节被刻意延迟展示。 Guided navigation over completeness.
下一步迭代方向
三个值得深入的方向：自动异常根因分析（当前设计仍需人工排查）、带置信区间的预测成本模型、以及告警路由至钉钉 / Slack，让决策闭环在仪表盘外也能完成。
关于云成本工具的普遍问题
FinOps 工具最常见的错误是优化数据完整性，而非决策速度。颜色语义的重要性被严重低估——当同一个绿色在不同面板里分别代表"节省"、"目标达成"和"数值下降"时，用户对界面的信任已经瓦解了。 Optimizing for decision speed, not data completeness.
